# Power Apps Code App — Project Scaffold

A portable blueprint for spinning up a new app with the **same stack, structure,
and DevOps loop** as the Enforcement AI dashboard.

Copy this file to the root of a new repo and follow the steps top-to-bottom. It
is **domain-agnostic** — fill in the placeholders marked `<<…>>`.

---

## 0. What this scaffold builds

A **Power Apps Code App** that:
- Runs as React + Vite + Tailwind inside the customer's Power Platform tenant.
- Reads/writes Dataverse through **one integration seam** (`src/services/dataverse.js`).
- Has a clean dev loop on mock data, and a one-command deploy via `pac code push`.
- Versions code on GitHub, publishes the app to Power Platform — **two separate pushes**.

If your new app does not need to live inside a Power Platform tenant (Entra-auth,
no-egress, Dataverse-backed), use a different scaffold — this one is overkill.

## 1. Required tools (one-time per machine)

| Tool | Why | Notes |
|---|---|---|
| **Node.js 22** (via `nvm`) | `pac code` breaks on older Node | `nvm install 22 && nvm use 22` |
| **Power Platform CLI** (`pac`) | Init + publish the Code App | Install via `dotnet tool install -g microsoft.powerapps.cli.tool` |
| **A Power Platform environment** with **Code Apps enabled** | Target tenant | Admin center → Environments → *your env* → Settings → Product → Features → **Power Apps Code Apps: On** |
| **A Power Apps license** in that tenant | To play the app | Per-user or per-app |
| **GitHub repo** | Source of truth | Create empty before `git init` |

> **macOS `pac` quirk:** keep the project in a folder path **without spaces**.
> A `pac` bug breaks on spaces. Example: use `~/Projects/MyApp`, not `~/My Apps/Foo`.

> **Auth vs code-command `pac` version mismatch (macOS):**
> - `pac auth create` → use `pac` **2.5.1** (newer crashes on macOS + .NET 10).
> - `pac code …` → use `pac` **2.8.1+**.
> Swap with `dotnet tool install -g microsoft.powerapps.cli.tool --version <ver> --force`.

## 2. Project skeleton

```
<<repo-name>>/
├─ CLAUDE.md                    # instructions for Claude Code in this repo
├─ README.md                    # short — quick start + link to ARCHITECTURE.md
├─ ARCHITECTURE.md              # end-to-end system + every Microsoft connection
├─ DEPLOY.md                    # the deploy TL;DR (cd → build → pac code push)
├─ .gitignore                   # blocks .env, dist/, client data file patterns
├─ .env.example                 # shows VITE_USE_MOCK + safe defaults
├─ package.json
├─ vite.config.js               # react + tailwind + @microsoft/power-apps-vite
├─ index.html
├─ power.config.json            # generated by `pac code init` — appId + envId live here
└─ src/
   ├─ main.jsx
   ├─ App.jsx                   # wrap your root with <PowerProvider>
   ├─ <<MainScreen>>.jsx        # composes the page + holds page state
   ├─ services/dataverse.js     # THE integration seam — mock now, swap for live
   ├─ lib/tokens.js             # design tokens (status / priority / nav meta)
   └─ components/               # presentational components
```

Two rules that make this scaffold work:

1. **One file talks to the backend.** `src/services/dataverse.js` is the *only*
   module that calls Dataverse, cloud flows, or Copilot. Components import
   `dataverseService` from it — nothing else.
2. **Mock is the default.** A `VITE_USE_MOCK=true` flag in `.env` swaps the
   real service for an in-memory mock. The dev loop never touches client data.

## 4. The integration seam — `src/services/dataverse.js`

Single source of all backend access. Three responsibilities:

```js
// 1. Contract — placeholder Dataverse logical names. Remap before going live.
export const ENTITY_CONTRACT = {
  <<table>>: {
    logicalName: 'cr1c4_<<table>>',
    fields: {
      id: 'cr1c4_<<table>>id',
      // …field map…
    },
  },
}

// 2. Mock — in-memory, deterministic, NO client data. Default in dev.
const mock = {
  async list<<Entity>>() { /* return seeded array */ },
  async update<<Entity>>(id, patch) { /* mutate seeded array */ },
  async <<callFlow>>({ replace }) { /* simulate ask-before-replace */ },
  async askAgent(prompt) { /* echo */ },
}

// 3. Live — wraps `pac code add-data-source`-generated typed services.
//          Filled in only when wiring to a sandbox/prod org.
const live = { /* same shape as mock, calls generated services */ }

const USE_MOCK = import.meta.env.VITE_USE_MOCK !== 'false'
export const dataverseService = USE_MOCK ? mock : live
```

**Required patterns to preserve:**

- **Same shape on both sides.** `mock` and `live` export an identical surface
  so swapping is one-line. Add a method to one → add it to the other.
- **Ask-before-replace for any data-changing flow.** First call `replace=false`;
  if the flow returns `exists`, surface a confirm modal, then call `replace=true`.
- **Never throw on optional fields.** If an upstream record is missing data,
  surface it in the UI (e.g. an amber "needs review" group) — don't drop it.

## 5. `power.config.json`

Generated by `pac code init`. After init, it looks like this — these are the
two IDs you'll reference everywhere:

```json
{
  "version": "1.0",
  "appId": "<<guid-from-pac-init>>",
  "appDisplayName": "<<App Display Name>>",
  "region": "prod",
  "environmentId": "<<env-guid>>",
  "buildPath": "./dist",
  "buildEntryPoint": "index.html",
  "localAppUrl": "http://localhost:3000",
  "connectionReferences": {
    "<<conn-guid>>": {
      "id": "/providers/Microsoft.PowerApps/apis/shared_commondataserviceforapps",
      "displayName": "Microsoft Dataverse",
      "dataSources": ["commondataserviceforapps"],
      "authenticationType": "Oauth",
      "dataSets": {}
    }
  }
}
```

The deployed play URL is derived from these:
`https://apps.powerapps.com/play/e/<environmentId>/a/<appId>`

## 8. Bootstrap — first time setting up the new repo

```bash
# 0. one-time setup
mkdir -p ~/Projects/<<RepoName>> && cd ~/Projects/<<RepoName>>   # NO SPACES in path
nvm use 22

# 1. scaffold Vite + React
npm create vite@latest . -- --template react

# 2. add the Power Apps + Tailwind deps
npm install @microsoft/power-apps lucide-react
npm install -D @microsoft/power-apps-vite @tailwindcss/vite tailwindcss

# 3. paste in vite.config.js, .gitignore, .env.example from this scaffold
#    and create src/services/dataverse.js with the mock seam.

# 4. git
git init && git add -A && git commit -m "scaffold"
gh repo create <<gh-org>>/<<RepoName>> --private --source=. --push

# 5. auth pac to your target environment
pac auth create --environment <<your-environment-url>> --deviceCode

# 6. turn this folder into a Power Apps Code App
pac code init --displayName "<<App Display Name>>"
# → wrap your root component in src/App.jsx with the generated <PowerProvider>.

# 7. create the Dataverse connection in Power Apps (Connections page),
#    then add each table you need:
pac code add-data-source -a shared_commondataserviceforapps -c <<conn-id>> -t <<table_logical_name>>

# 8. local dev (mock data)
npm run dev
```

## 9. The DevOps loop — every change after bootstrap

```bash
cd ~/Projects/<<RepoName>>
nvm use 22

# 1. SAVE to GitHub (history / backup)
git add -A
git commit -m "describe what changed"
git push

# 2. PUBLISH to the live app
npm run build
pac code push
```

Watch for **"App pushed successfully. You can play your app at https://…"**.

| Command | Where it goes | What it does |
|---|---|---|
| `git push`        | **GitHub**         | Versions your code. Does **not** change the live app. |
| `pac code push`   | **Power Platform** | Publishes the live app. Does **not** touch GitHub. |

Decoupling these on purpose: WIP can land on GitHub without going live, and a
hotfix can ship to the tenant without waiting on a PR review.

## 10. Going from mock → live data (one-time per app)

1. In **Power Apps → Connections**, create a **Microsoft Dataverse** connection.
   Note its connection id. (`pac` uses existing connections; it can't create them.)
2. Generate typed services for each table:
   ```bash
   pac code add-data-source -a shared_commondataserviceforapps -c <<conn-id>> -t <<table>>
   ```
   This writes to `./generated/services/` — commit it.
3. Implement the `live` object in `src/services/dataverse.js` using those
   generated services. Keep the **same shape** as `mock`.
4. Flip the toggle against a **sandbox** org first:
   ```bash
   echo "VITE_USE_MOCK=false" >> .env
   npm run dev
   ```
5. `npm run build && pac code push` once it works end-to-end.

Never point dev at a production org. The sandbox boundary is the safety net.

## 11. Red-line rules — same for every app built this way

These belong verbatim in the new repo's `CLAUDE.md`.

- **No real client data in the repo.** Ever. Develop on mock or a sandbox with
  synthetic data. `.gitignore` enforces it as a backstop, not as the primary
  defense — the primary defense is "don't put it there."
- **No secrets in the repo.** No connection strings, API keys, tenant URLs with
  embedded creds. `.env` is gitignored.
- **One backend seam.** Only `src/services/dataverse.js` talks to Dataverse / a
  flow / Copilot. Adding a new component? It imports `dataverseService` —
  never a connector directly.
- **Ask-before-replace.** Any flow that overwrites data must first call
  `replace=false`, surface a confirm modal on `exists`, then call `replace=true`.
- **No-egress is a feature, not a bug.** Source plane (your laptop, GitHub) and
  data plane (deployed app + tenant) are separate. If a request would force
  client data through the source plane, push back.

## 12. Troubleshooting (copy from DEPLOY.md of the original repo)

- **`403 CodeAppOperationNotAllowedInEnvironment`** — Code Apps is off in that
  environment's feature settings, or the change is still propagating (~10–15 min).
- **`pac: command not found`** — open a new terminal or `nvm use 22` first.
- **`pac` hangs / "Broken pipe"** — don't pipe its output (no `| tail`). Ctrl-C
  and rerun. Make sure `nvm use 22` ran in this terminal.
- **`pac auth create` crashes on macOS** — downgrade to `pac` 2.5.1 just for the
  auth step, then back to 2.8.1+ for `pac code` commands.
- **Browser blocks Local Play URL** — Chrome/Edge may block local-network access
  since Dec 2025; grant it when prompted. Always open in the same browser
  profile signed into the tenant.
